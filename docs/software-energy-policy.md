# 位置精度・省電力ポリシー

## 1. 優先順位

ソフトウェア設計では次の順序を採用する。

1. 山笠の公開位置の精度と連続性
2. 主催者スマートフォン1台あたりの電池消費抑制
3. ネットワークアクセスとバックグラウンド起動回数の削減
4. 開催・非開催判定
5. 開催状態が切り替わる時刻の即時性

開催・非開催の切り替えタイミングは厳密でなくてよい。開催判定を高速化するために、全端末を頻繁に起動・通信させてはならない。

本書は `software-design.md` の省電力、送信競合抑制、presence、開催判定に関する方針を具体化し、矛盾する箇所について本書を優先する。

## 2. 基本原則

ほとんどの端末は、対象BLEタグをOSのバックグラウンド機構で監視し、ローカル状態だけを更新する。

各更新周期に全端末がサーバーへ問い合わせる方式は採用しない。

```text
全端末
  └── BLE検出とローカル選出のみ

少数の候補端末
  └── 位置を評価
        ├── 十分なキャッシュ位置があれば利用
        └── 不十分なら単発の高精度測位
              ↓
           POST 1回
```

省電力化は、少量の通信データを削ることではなく、次の回数を減らすことで達成する。

- バックグラウンドからのアプリ起動
- セルラー／Wi-Fi無線の起動
- 新規位置測位
- 高精度GNSSの起動
- 定期タイマー処理

## 3. 分散送信

### 3.1 通信を伴わない候補選出

端末は時間スロットと匿名セッションIDからローカルに候補順位を計算する。

```text
score = HMAC(
  session_secret,
  yamakasa_id | slot_id | relay_session_id
)
```

この計算のためにサーバーへアクセスしない。

### 3.2 少数重複を許容する

厳密な単一リーダー選出は行わない。排他制御のために全端末がGETするより、少数端末による重複POSTを許容する方が全体消費を抑えられる。

初期値：

- 位置更新スロット：30秒
- 1スロットの期待送信端末数：2台
- 最大候補階層：3段階
- 同一端末の担当クールダウン：5分

候補階層：

```text
0〜8秒    第一候補群
10〜18秒  第二候補群
20〜28秒  第三候補群
```

第二・第三候補は、サーバー確認のためのGETを行わない。候補確率を低く設定し、通常は第一候補群だけが送信する。結果として同じスロットで1〜3件届くことは許容し、バックエンドが最良位置を採用する。

### 3.3 候補確率

端末数を厳密に知る必要はない。過去に受信した低頻度の推定参加者数があれば利用し、なければ保守的な固定確率を使う。

暫定10人を開催閾値とする初期運用では、開催中の期待送信数を約2台とする。

```text
candidate_probability = clamp(2 / estimated_device_count, min, max)
```

推定端末数が不明な場合は、各端末が約20%の確率で第一候補になる設定から実測調整する。

## 4. 位置取得

### 4.1 精度を省電力より下位に置かない

送信担当に選ばれた端末は、古い低精度位置を無条件に送信してはならない。

採用順序：

1. OSキャッシュ位置が20秒以内かつ水平精度20m以内なら採用
2. それ以外は単発位置取得を開始
3. 水平精度15〜25mを目標に最大8秒待つ
4. 8秒以内に50m以内が得られれば送信
5. 50mを超える位置は、他候補が存在しない場合のフォールバックとして信頼度を下げて扱う

常時位置監視は行わない。高精度測位は選出された端末だけが単発で行い、取得後直ちに停止する。

### 4.2 複数レポートを精度向上に使う

同じスロットで複数レポートが届いた場合、単なる重複として全破棄せず、位置精度向上へ利用する。

- 水平精度が悪い報告を除外
- 最大速度と経過時間から不可能な位置を除外
- 相互距離が大きい外れ値を除外
- 残った位置を精度の逆分散で重み付け
- 道路スナップ後の経路連続性も評価

位置精度のため、期待送信数を完全に1台へ絞らず、2台程度の独立測位を許容する。

## 5. ネットワークアクセス

### 5.1 禁止する通信

- 各スロット前の `GET relay-state`
- 20〜30秒ごとの全端末presence
- 排他制御だけを目的とするポーリング
- 全端末へのスロット更新プッシュ
- BLE受信ごとの即時アップロード

### 5.2 許可する通信

- 選出端末による位置レポートPOST
- 位置レポートPOSTに同梱するpresence更新
- BLE検出開始時の低頻度presence
- 数分から十数分単位の状態更新
- アプリがフォアグラウンドになった際の設定同期

位置レポートに次を同梱し、別々の通信を避ける。

- 匿名セッションID
- タグ最終検出時刻
- 検出タグ数
- RSSI統計
- バッテリー状態区分
- 低電力モードの有無

## 6. 開催・非開催判定

### 6.1 判定基準

暫定的に10台を境界とする。

- 推定有効端末数が10台以上：`ACTIVE`
- 推定有効端末数が10台未満：`INACTIVE`

ただし、判定時刻の厳密性は求めない。

### 6.2 低頻度presence

端末は次のタイミングだけpresenceを更新する。

- タグを新たに検出したとき
- 自端末が位置レポートを送るとき
- 最後のpresenceから5〜10分以上経過し、OSが自然にアプリを起動したとき
- アプリがフォアグラウンドになったとき

presenceだけのために正確な定期タイマーでアプリを起動しない。

### 6.3 長いリースとヒステリシス

初期値：

- presenceリース：10分
- `INACTIVE → ACTIVE`：推定10台以上を3分以上観測
- `ACTIVE → INACTIVE`：推定10台未満を15分以上観測

端末数は正確なリアルタイム人数ではなく、直近の匿名セッション集合から算出する概算値とする。

開催状態が数分から十数分遅れて切り替わっても許容する。位置更新処理は開催状態に依存させず、タグを検出している候補端末が存在する限り継続する。

## 7. 道路スナップと進行方向

省電力化によって位置点数を過度に減らさない。30秒ごとに1〜2件の品質のよい位置を得ることを目標とする。

バックエンドは次の順で処理する。

1. 同一スロットの位置統合
2. 過去数分の平滑化軌跡を生成
3. Google Roads APIで近傍道路へスナップ
4. 前回の道路区間との連続性を検証
5. 直近60〜120秒の道路上移動から進行方向を推定

進行方向は単一スマートフォンのheadingやcourseを直接採用せず、道路スナップ済み位置時系列から求める。

方向更新条件：

- 道路上の移動距離が位置誤差を十分上回る
- 2点だけでなく3点以上が同じ道路系列上で整合する
- 不自然なUターンや並行道路へのジャンプがない

方向信頼度は `HIGH`、`MEDIUM`、`LOW` を公開する。

## 8. 初期パラメータ

| 項目 | 初期値 |
|---|---:|
| 位置更新スロット | 30秒 |
| 期待位置レポート数 | 2件／スロット |
| キャッシュ位置の最大経過時間 | 20秒 |
| キャッシュ位置の目標精度 | 20m以内 |
| 単発測位タイムアウト | 8秒 |
| 通常採用上限 | 50m |
| 同一端末クールダウン | 5分 |
| presenceリース | 10分 |
| ACTIVE開始 | 10台以上を3分 |
| INACTIVE開始 | 10台未満を15分 |
| 進行方向推定窓 | 60〜120秒 |

これらは実機で、位置誤差、1時間当たりの電池消費、位置レポート重複数、道路スナップ誤り率を計測して調整する。

## 9. 成功指標

- 観客向け道路上位置の中央値誤差
- 95パーセンタイル位置誤差
- 誤った並行道路へのスナップ率
- 進行方向の正答率と信頼度較正
- 端末1台当たりの1時間の位置測位回数
- 端末1台当たりの1時間のネットワーク起動回数
- イベント1時間当たりの位置レポート総数
- 1時間当たりのバッテリー消費率
- 開催状態の切り替え遅延

開催状態の切り替え遅延は監視するが、位置精度と電池消費より優先しない。
