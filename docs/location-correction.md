# タグ・中継端末間の位置補正設計

## 1. 目的

主催者スマートフォンが送信する位置は、山笠そのものではなく、山笠周囲にいる端末の位置である。本設計は、複数端末と4個のBLEタグから得られる情報を使い、観客へ公開する山笠の中心位置を推定する。

本書は [ソフトウェア設計](software-design.md) のバックエンド位置統合を詳細化する補助設計である。識別子、測位品質基準、API URI、開催判定、道路補正との責務が異なる場合は `software-design.md` を優先する。

## 2. 設計原則

- BLEのRSSIを距離へ直接変換しない
- 単一端末の位置を山笠位置として無条件に採用しない
- 複数端末の位置、測位精度、BLE観測品質を統合する
- タグ配置の既知オフセットは、位置品質が十分な場合だけ補助的に使う
- スマートフォンのheading、bearing、端末姿勢を山笠の向きとして使わない
- 道路スナップ前に外れ値を除去し、道路スナップ後に進行方向を推定する
- 補正によって元の測位精度以上の確度を装わない

## 3. 入力データ

位置レポートは、少なくとも次を含む。

```json
{
  "reportId": "uuid",
  "relaySessionId": "rotating-id",
  "resourceId": "01af34c9812de704",
  "slotId": 123456,
  "observedAt": "timestamp",
  "location": {
    "latitude": 0.0,
    "longitude": 0.0,
    "horizontalAccuracyM": 12.0,
    "ageMs": 4500,
    "source": "fresh-single-fix"
  },
  "tagObservations": [
    {
      "nodeId": "front",
      "lastSeenAgeMs": 900,
      "sampleCount": 6,
      "rssiMedian": -65
    },
    {
      "nodeId": "left",
      "lastSeenAgeMs": 1200,
      "sampleCount": 4,
      "rssiMedian": -70
    }
  ]
}
```

`resourceId`はBLE広告の`resource_id`と同じ64-bit値の16進文字列表現とする。

## 4. 補正処理の全体順序

```text
各端末のGNSS位置
  ↓
位置時刻・精度・BLE近傍性の検証
  ↓
同一スロット内の外れ値除去
  ↓
複数端末位置の重み付き統合
  ↓
条件を満たす場合だけタグ固定オフセット補正
  ↓
生の山笠中心推定位置
  ↓
Google Roads APIによる道路補正
  ↓
道路補正済み時系列から進行方向推定
```

道路スナップは、端末とタグの位置関係を補正する機能ではない。端末群から推定した山笠中心位置を、近傍道路へ整合させる後段処理である。

## 5. BLE近傍性の評価

### 5.1 用途

BLE観測は、端末が山笠の近傍にいる確からしさと、位置レポートの重み付けに使う。

評価要素：

- 最終検出からの経過時間
- 直近期間の検出回数
- 検出した`node_id`の数
- RSSI中央値
- RSSIのばらつき
- 人体や装飾による一時的遮蔽に対する継続性

### 5.2 RSSIの扱い

RSSIは距離の直接推定には使わない。

採用する用途：

- 単発で極端に弱い観測の重みを下げる
- 複数タグを継続検出したレポートの信頼度を上げる
- 同一端末内で観測品質が急激に悪化したことを検出する

採用しない用途：

- RSSIからメートル単位の距離を算出する
- 3個以上のタグRSSIによる三辺測量
- RSSI最大のタグ位置を端末位置として採用する

山笠では、人体、人形、板、布、LED配線、雨、端末の保持位置によってRSSIが大きく変動するためである。

## 6. 複数端末位置の統合

### 6.1 外れ値除去

同一スロットの位置レポートを次の順で検証する。

1. 水平精度50m超を原則除外
2. BLEタグを送信直前に有効検出していないレポートを除外
3. 前回推定位置から想定最高速度を超えて移動した点を除外
4. 他端末群の位置から大きく離れた点を除外
5. 時刻が古い点の重みを下げる

外れ値判定では、GNSSの`horizontalAccuracyM`を考慮し、精度円が重なる位置を単純に異常としない。

### 6.2 重み

各レポートの基礎重みは、次を組み合わせる。

```text
weight =
  location_accuracy_weight
  * location_freshness_weight
  * ble_freshness_weight
  * ble_stability_weight
  * detected_node_count_weight
```

初期方針：

- 水平精度が高いほど重くする
- 位置とBLE観測が新しいほど重くする
- 2個以上のタグを安定検出したレポートをやや重くする
- 1個だけを弱く単発検出したレポートを軽くする
- RSSIの強さだけで支配的な重みを与えない

### 6.3 統合方法

初期版は、外れ値除去後に以下を比較する。

1. 最良精度点を採用
2. 精度の逆分散による重み付き中心
3. ロバストな幾何中央値

端末位置が山笠の周囲へ偏っている場合、単純平均は山笠中心からずれる。実地データで、最良精度点、重み付き中心、幾何中央値を比較し、誤差が最も小さい方法を採用する。

## 7. タグ固定オフセット補正

### 7.1 目的

4個のタグは山笠中心から既知の位置にある。`node_id`ごとの取り付け位置を、山笠ローカル座標として管理できる。

```text
front: (+x,  0)
rear:  (-x,  0)
left:  ( 0, +y)
right: ( 0, -y)
```

ただし、端末はタグ位置に存在するとは限らない。タグを数m離れて検出するため、`node_id`だけを使って固定距離を機械的に引く補正は行わない。

### 7.2 初期版での利用

初期版では固定オフセットを、次の用途に限定する。

- 複数端末の検出パターンから位置偏りを評価する
- 同一方向タグだけを検出したレポート群が一方向に偏る場合の弱い補正候補
- 道路スナップ候補が複数ある場合の連続性評価
- 実測データを用いたオフライン分析

オンライン補正へ採用する条件：

- タグの年次配置位置が測量・登録されている
- 山笠の向きが道路時系列から十分な信頼度で推定されている
- 複数端末レポートが存在する
- GNSS精度とレポート分散が補正量より十分小さい
- 補正後の位置が道路連続性と矛盾しない

位置精度が10〜30mで、タグオフセットが数mの場合、固定補正の効果は限定的である。初期導入では、複雑なオフセット補正より、外れ値除去と複数端末統合を優先する。

## 8. 毎年変わるタグ配置への対応

山笠の人形や外装は毎年変わるため、`node_id`だけで物理座標を固定しない。

年次設定として次を保持する。

```text
resource_installation
├── resource_id
├── season
├── node_id
├── local_offset_x_m
├── local_offset_y_m
├── local_offset_z_m
├── installation_notes
└── calibration_version
```

- `resource_id`は年をまたいで同一でもよい
- タグの物理位置は`season`と`calibration_version`で管理する
- 位置を測定していない年は固定オフセット補正を無効にする
- 配置変更後は実測試験を行い、過年度の補正値を流用しない

## 9. 採用しない補正

初期版では以下を採用しない。

- BLE RSSI距離推定
- BLE RSSI三辺測量
- スマートフォンの磁気方位を山笠方向として利用
- スマートフォンの加速度・ジャイロによる山笠姿勢推定
- 単一タグ検出だけによる固定ベクトル補正
- 単一端末位置だけから山笠中心を逆算
- 道路スナップ結果を常に正解として強制採用

これらは、実環境で再現性が確認できた場合に限り、別途実験機能として評価する。

## 10. 出力と信頼度

バックエンドは、公開位置とともに内部的な補正情報を保持する。

```json
{
  "position": {
    "latitude": 0.0,
    "longitude": 0.0,
    "accuracyM": 16.0,
    "source": "road-snapped",
    "updatedAt": "timestamp"
  },
  "estimation": {
    "reportCount": 2,
    "acceptedReportCount": 2,
    "detectedNodeCount": 3,
    "fusionMethod": "weighted-center",
    "tagOffsetApplied": false,
    "confidence": "MEDIUM"
  }
}
```

観客向けAPIに内部診断値をすべて公開する必要はない。管理・検証用途として保持する。

信頼度は少なくとも次を考慮する。

- 採用レポート数
- レポート間の分散
- 水平精度
- BLE観測鮮度と検出タグ数
- 道路補正の連続性
- 固定オフセット補正の有無と校正バージョン

## 11. 検証

実地試験では、山笠中心の基準位置を別手段で記録し、各推定方式と比較する。

測定条件：

- 前後左右・斜めに端末が偏った状態
- 10人、20人、50人程度の参加端末数
- 人形の全開・全閉・途中
- 電飾OFF・ON・最大輝度・演出中
- 停止・低速移動・走行相当の移動
- 晴天・雨天
- iPhone・Android・混在

評価指標：

- 生端末位置の誤差
- 外れ値除去後の誤差
- 複数端末統合後の誤差
- タグ固定オフセット適用前後の誤差
- 道路スナップ後の誤差
- 誤った道路へのスナップ率
- 位置誤差50・90・95パーセンタイル
- 補正による改善率と悪化率

固定オフセット補正は、統計的に改善し、悪化率が許容範囲内であることを確認するまで本番有効化しない。

## 12. 初期実装方針

初期版：

1. BLE近傍性の検証
2. GNSS位置の品質評価
3. 同一スロット内の外れ値除去
4. 最良精度点またはロバスト統合
5. Google Roads APIによる道路補正
6. 位置時系列から進行方向推定

後続検証：

1. 年次タグ配置座標の登録
2. タグ検出パターンによる偏り分析
3. 固定オフセット補正のオフライン評価
4. 効果が確認された場合だけオンライン適用

**初期版の主要価値は、BLEで端末が山笠近傍にいることを確認し、複数端末の高品質なGNSS位置を統合することにある。RSSI測距やタグオフセット補正は主要経路に置かない。**
