# ハードウェア設計

## 1. 設計方針

山笠1基に4個の独立したBLEタグを搭載する。

各タグはGPSやLTEを持たず、固定の山笠IDと方向別タグIDをBLE広告する。周囲の主催者スマートフォンが広告を検出し、端末群で位置取得・送信を分担する。

4タグ化の目的は通信距離の延長ではなく、山笠の巨大な形状、板・布、人形、LED配線、人体による方向別の遮蔽を相互補完することである。

## 2. タグ構成

各タグは同一ハードウェアとし、書き込む`node_id`だけを変える。

```text
CR2032
  │
  ├── 電源デカップリング
  │     ├── 100nF
  │     └── 10〜47µF 低漏れコンデンサー
  │
  └── BLE SoC
        ├── 32MHz水晶
        ├── 任意の32.768kHz水晶
        ├── 2.4GHzアンテナ
        ├── 電池電圧測定
        └── 書き込み・デバッグパッド
```

### 2.1 BLE SoC

要求：

- BLE 5.x対応
- 非接続広告
- 0dBmから+8dBm程度まで送信出力を設定可能
- CR2032直結可能な動作電圧
- AES/HMAC相当の認証処理を実装可能
- 低消費電力スリープ

初期候補はNordic nRF52833相当とする。量産時には必要なメモリ、価格、入手性、認証済みモジュールの有無を比較して確定する。

### 2.2 電源

CR2032からSoCへ直接給電する。

- レギュレーターは原則搭載しない
- SoC直近に100nFを配置する
- 電池直近に10〜47µFの低漏れコンデンサーを配置する
- 最大送信出力時のピーク電流でブラウンアウトしないことを確認する
- 低温、電池末期、電飾点灯時のノイズ環境で再起動しないことを試験する

電池寿命は1週間が要件であるため、平均消費電流の最小化より、安定した広告受信率と電池接点の信頼性を優先する。

### 2.3 電池ホルダー

山笠の振動を考慮し、一般的な弱い板バネだけに依存しない。

推奨方式：

- 横挿入ロック式ホルダー
- 上蓋が電池を押さえる構造
- 電池上面に薄い弾性パッドを配置

電池交換時に工具を必要としないことより、防水再現性と接点保持を優先してよい。

## 3. アンテナ設計

### 3.1 方針

4タグそれぞれが前・後・左・右の担当方向へ安定して広告できればよく、単一タグで完全な無指向性を実現する必要はない。

タグのアンテナ側を山笠の外側へ向け、電池、GND、木材、布、LED配線、金属装飾を山笠側へ寄せる。

```text
山笠側

[CR2032][BLE SoC][アンテナ keep-out]  → 担当方向
```

### 3.2 アンテナkeep-out

アンテナ周辺では以下を避ける。

- 銅箔
- 電池
- 金属ネジ
- 金属ブラケット
- LED配線束
- 大型金属装飾
- 吸水した布への密着

基板アンテナまたはFPCアンテナの最終選定は、実筐体・実設置状態での測定後に行う。

### 3.3 初期無線設定

| 項目 | 初期値 |
|---|---:|
| PHY | BLE 1M |
| 広告間隔 | 250ms |
| 送信出力 | 0dBm |
| 接続受付 | 無効 |
| 運用LED | 無効 |

試験結果により、タグ単位で-4、0、+4、+8dBmを選択する。

全タグを最初から最大出力にしない。過大な到達範囲は、山笠から離れた端末を位置送信候補に残し、位置品質を低下させる可能性がある。

## 4. 広告プロトコル

### 4.1 ペイロード

```text
protocol_version   1 byte
yamakasa_id        4 bytes
node_id            1 byte
boot_counter       2 bytes
sequence           2 bytes
battery_state      1 byte
auth_tag           8 bytes
```

合計19バイトを想定し、Manufacturer Specific Dataに収める。

### 4.2 ID

- `yamakasa_id`: 山笠を表す固定公開ID
- `node_id`: `front`, `rear`, `left`, `right`

山笠の位置は意図的に公開するため、IDローテーションは行わない。

### 4.3 認証

固定IDの秘匿ではなく、偽タグとリプレイの防止を目的とする。

```text
auth_tag = Truncate(MAC(
  tag_secret,
  protocol_version |
  yamakasa_id |
  node_id |
  boot_counter |
  sequence |
  battery_state
))
```

秘密鍵はタグごとに異なるものを使用し、サーバー側で管理する。

アプリは広告を収集してサーバーへ渡し、最終的な認証をサーバーで行う。オフライン検証が必要になった場合のみ、アプリ配布鍵または署名方式を追加検討する。

### 4.4 シーケンス

`sequence`は一定時間ごとに増加させる。

広告パケットごとに増加させる必要はない。例えば5〜30秒ごとの更新で十分である。これにより同じ広告を複数回受信しても、アプリ側で安定検出回数として扱える。

`boot_counter`は不揮発領域へ保存し、再起動ごとに増加させる。書き込み寿命を考慮し、異常な再起動ループを防ぐ。

## 5. ファームウェア設計

### 5.1 状態遷移

```text
BOOT
  ↓
設定・鍵・boot_counter読込
  ↓
広告ペイロード生成
  ↓
ADVERTISING
  ├── sequence更新
  ├── 電池状態更新
  ├── MAC再計算
  └── watchdog監視
```

通常運用では広告を停止しない。

夜間に山笠が街中へ停車している期間も、位置は公開対象であるため、24時間広告を継続する。

### 5.2 電池状態

CR2032の残量を高精度パーセントとして表現せず、運用判断に必要な段階値にする。

- `GOOD`
- `LOW`
- `REPLACE`

送信時の電圧降下を避けるため、無線送信直後ではなく安定したタイミングで測定する。

### 5.3 障害復旧

- watchdogを有効化する
- ブラウンアウト後に自動復旧する
- 設定破損時は安全な既定値で広告するか、明確なmaintenance IDを送信する
- LED点滅による恒常的な障害通知は行わない

## 6. 筐体設計

### 6.1 構造

```text
上蓋
├── Oリング
├── 電池押さえパッド
├── CR2032
├── 基板
└── 下ケース・固定耳
```

要求：

- IP67相当
- 電池交換式
- ポリカーボネート、ASA、UV安定化ABS等の非導電性材料
- アンテナ前方に金属部品を置かない
- 金属ネジを使う場合はアンテナから離す
- 結露、水滴、振動に耐える

### 6.2 目標寸法

- 外形：30〜45mm
- 厚さ：10〜15mm
- 重量：10〜20g

最小化より、アンテナkeep-out、防水シール幅、電池保持、固定耳の強度を優先する。

## 7. 4タグの論理統合

4タグは物理的には独立しているが、アプリとバックエンドでは1つの山笠として扱う。

```text
Yamakasa 001
├── front
├── rear
├── left
└── right
```

アプリはどれか1タグを有効に検出すれば、その山笠の位置送信候補になれる。

複数タグを検出した場合、送信回数を増やさず、次の品質情報としてまとめる。

- 検出タグ数
- 各タグの最終検出時刻
- 各タグの受信回数
- RSSI中央値

RSSIから距離を精密推定することは目的としない。

## 8. 製造・設定

各タグへ以下を書き込む。

- protocol version
- yamakasa ID
- node ID
- tag secret
- initial boot counter
- RF calibration/configuration

外装にも前・後・左・右を識別する恒久ラベルを付ける。

QRコード等でタグの製造情報と管理画面を関連付けることを推奨するが、BLE広告の公開IDと秘密鍵を同じラベルへ印字しない。

## 9. 設計上の未確定事項

以下は実機検証で確定する。

- PCBアンテナかFPCアンテナか
- 各タグの最適送信出力
- 広告間隔250msが必要か、500msで十分か
- 32.768kHz水晶の必要性
- 筐体寸法とスペーサー長
- 前後左右の最終取り付け高さ
- LED点灯パターンによる2.4GHz受信率への影響
