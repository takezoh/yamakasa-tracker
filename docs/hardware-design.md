# ハードウェア設計

## 1. 設計方針

山笠1基に4個の独立したBLEタグを搭載する。各タグはGPSやLTEを持たず、同じ`resource_id`と方向別の`node_id`をBLE広告する。

4タグ化の目的は通信距離の延長ではなく、山笠の巨大な形状、板・布、人形、LED配線、人体による方向別遮蔽を相互補完することである。

**ハードウェアは山笠の外観を損なってはならない。** 外観保全は無線性能、防水、保守性、コストより上位の制約とする。

## 2. タグ構成

```text
CR2032
  │
  ├── 100nF
  ├── 10〜47µF低漏れコンデンサー
  │
  └── BLE SoC
        ├── 32MHz水晶
        ├── 任意の32.768kHz水晶
        ├── PCBアンテナまたはFPCアンテナ接続
        ├── 電池電圧測定
        └── 書き込み・デバッグパッド
```

要求：

- BLE 5.x、非接続広告
- 0〜+8dBm程度の出力設定
- CR2032直結可能
- MAC生成に必要な暗号機能
- 低消費電力スリープ
- 隠蔽空間へ収められる形状

初期候補はNordic nRF52833相当とするが、量産時には実装面積、価格、入手性、認証済みモジュールの有無を比較して確定する。

## 3. 識別子設計

### 3.1 resource_id

追跡対象の識別子は全層で`resource_id`に統一する。

- 論理型：unsigned 64-bit
- BLE：8-byte big-endian binary
- API・JSON・ログ：16桁の小文字16進文字列
- DB：`BIGINT`相当または8-byte binary

`yamakasa_id`、`beacon_id`、`resource_ref`などの別名を設けない。

### 3.2 node_id

`node_id`は同一`resource_id`に属する4タグを区別する。

- `front`
- `rear`
- `left`
- `right`

### 3.3 広告ペイロード

```text
protocol_version   1 byte
resource_id        8 bytes
node_id            1 byte
boot_counter       2 bytes
sequence           2 bytes
battery_state      1 byte
auth_tag           8 bytes
```

合計23バイトをManufacturer Specific Dataへ収める。

### 3.4 認証

```text
auth_tag = Truncate(MAC(
  tag_secret,
  protocol_version |
  resource_id |
  node_id |
  boot_counter |
  sequence |
  battery_state
))
```

`resource_id`は公開固定IDでありローテーションしない。認証はID秘匿ではなく、偽タグとリプレイ防止を目的とする。

## 4. 電源

- CR2032からSoCへ直接給電
- レギュレーターは原則搭載しない
- SoC直近に100nF
- 電池直近に10〜47µF低漏れコンデンサー
- 最大出力時、低温、電池末期でもブラウンアウトしないことを確認
- 電池ホルダーは横挿入ロック式、上蓋押圧式など振動に強い方式
- 最低168時間動作

## 5. アンテナ

### 5.1 基本方針

プロトタイプは一体型BLEモジュールまたは評価基板の内蔵PCBアンテナを使う。製品化も小型一体型を基本とする。

外観を保った隠蔽配置で必要な検出率を満たせない方向のみ、短い同軸ケーブル付きFPCアンテナ分離型を採用する。

### 5.2 構成候補

1. 一体型PCBアンテナ
2. 一体型チップアンテナ
3. 短い同軸付きFPCアンテナ
4. 配置空間に合わせた方向別基板派生

分離アンテナは局所的な配置自由度を得るための手段であり、中央から数メートル引き回す設計にはしない。ケーブル長は10〜30cmを目標とし、長くても50cm程度を上限候補とする。

### 5.3 keep-out

アンテナ周辺では以下を避ける。

- 銅箔、電池
- 金属ネジ、金属ブラケット
- LED配線束
- 大型金属装飾
- 吸水した布への密着

アンテナ形式と整合は、裸基板ではなく山笠へ隠蔽した最終状態で評価する。

## 6. 初期無線設定

| 項目 | 初期値 |
|---|---:|
| PHY | BLE 1M |
| 広告間隔 | 250ms |
| 送信出力 | 0dBm |
| 接続受付 | 無効 |
| 運用LED | 無効 |

試験結果により各タグを-4、0、+4、+8dBmから選択する。全タグを最初から最大出力にしない。

## 7. ファームウェア

```text
BOOT
  ↓
設定・鍵・boot_counter読込
  ↓
広告ペイロード生成
  ↓
ADVERTISING
  ├── sequence更新
  ├── 電池状態更新
  ├── MAC再計算
  └── watchdog監視
```

- 通常運用では24時間広告を継続
- `sequence`は5〜30秒程度ごとに更新
- `boot_counter`は再起動ごとに増加
- 電池状態は`GOOD`、`LOW`、`REPLACE`
- watchdog、ブラウンアウト復旧、設定破損時の安全動作
- 運用中LEDなし

## 8. 筐体

筐体は独立した見える製品ではなく、山笠の既存構造内へ完全に隠蔽される内部部品として設計する。

- IP67相当
- 電池交換式
- 低光沢・非導電性材料
- 山笠の外形から突出しない
- 人形の全可動域で露出・干渉しない
- 外部に見えるLED、ボタン、ロゴ、ラベルなし
- 固定部と脱落防止部も隠蔽
- 取り外し後に痕跡を残さない

形状は円形や正方形に固定せず、隠蔽空間に応じて薄型長方形、細長い形状、方向別筐体を許容する。

## 9. 製造・設定

各タグへ以下を書き込む。

- protocol version
- resource ID
- node ID
- tag secret
- initial boot counter
- RF configuration

保守ラベルとQRコードは取り外した時だけ見える面に付ける。秘密鍵をラベルへ印字しない。

## 10. 未確定事項

- SoCまたは認証済みモジュールの最終選定
- PCB、チップ、FPCアンテナの方向別採否
- 最適送信出力と広告間隔
- 32.768kHz水晶の必要性
- 筐体寸法と電池ホルダー
- 毎年の装飾に応じた最終配置
- 電飾点灯パターンによる受信率への影響
