# ハードウェア設計

## 1. 設計方針

山笠1基に4個の独立したBLEタグを搭載する。

各タグはGPSやLTEを持たず、固定の山笠IDと方向別タグIDをBLE広告する。周囲の主催者スマートフォンが広告を検出し、端末群で位置取得・送信を分担する。

4タグ化の目的は通信距離の延長ではなく、山笠の巨大な形状、板・布、人形、LED配線、人体による方向別の遮蔽を相互補完することである。

### 1.1 最上位設計制約

**ハードウェアは山笠の外観を損なってはならない。**

この制約は無線性能、防水、保守性、コスト、既製部品の利用より上位に置く。

- タグ本体、アンテナ、筐体、固定具、ラベルを通常の観覧位置から見せない
- 山笠の輪郭、造形、色、質感、電飾、人形の演出を変えない
- 外装面への露出設置、外形からの突出、目立つスペーサーを基本設計に含めない
- 外観を満たせない場合は、意匠側を変更せず、基板、アンテナ、筐体、タグ数、配置、ソフトウェアを再設計する

したがって、物理設計は既定サイズの筐体を配置するのではなく、山笠内部・裏面に存在する非視認空間を先に測定し、その空間へ収まる構成を逆算する。

## 2. タグ構成

各タグは同一の論理機能を持ち、書き込む`node_id`だけを変える。物理形状は配置空間に応じて、同一基板または方向別の筐体・アンテナ派生を許容する。

```text
CR2032
  │
  ├── 電源デカップリング
  │     ├── 100nF
  │     └── 10〜47µF 低漏れコンデンサー
  │
  └── BLE SoC
        ├── 32MHz水晶
        ├── 任意の32.768kHz水晶
        ├── 2.4GHzアンテナ、またはFPCアンテナ接続
        ├── 電池電圧測定
        └── 書き込み・デバッグパッド
```

### 2.1 BLE SoC

要求：

- BLE 5.x対応
- 非接続広告
- 0dBmから+8dBm程度まで送信出力を設定可能
- CR2032直結可能な動作電圧
- AES/HMAC相当の認証処理を実装可能
- 低消費電力スリープ
- 隠蔽空間へ収められる小型パッケージまたはモジュール

初期候補はNordic nRF52833相当とする。量産時には必要なメモリ、価格、入手性、実装面積、認証済みモジュールの有無を比較して確定する。

### 2.2 電源

CR2032からSoCへ直接給電する。

- レギュレーターは原則搭載しない
- SoC直近に100nFを配置する
- 電池直近に10〜47µFの低漏れコンデンサーを配置する
- 最大送信出力時のピーク電流でブラウンアウトしないことを確認する
- 低温、電池末期、電飾点灯時のノイズ環境で再起動しないことを試験する

電池寿命は1週間が要件であるため、平均消費電流の最小化より、安定した広告受信率と電池接点の信頼性を優先する。ただし、大型電池や厚いホルダーによって隠蔽不能になる場合は、薄型ホルダー、基板配置、電池とアンテナの分離を再設計する。

### 2.3 電池ホルダー

山笠の振動を考慮し、一般的な弱い板バネだけに依存しない。

候補：

- 薄型横挿入ロック式ホルダー
- 筐体または内側固定板が電池を押さえる構造
- 電池上面に薄い弾性パッドを配置

工具不要より、防水再現性、接点保持、隠蔽可能な厚さを優先する。

## 3. アンテナ設計

### 3.1 方針

4タグそれぞれが前・後・左・右の担当方向へ安定して広告できればよく、単一タグで完全な無指向性を実現する必要はない。

ただし、アンテナを担当方向の外面へ露出させることは前提にしない。既存の非金属外装、板、布、装飾の裏側から必要なリンクバジェットを確保する。

### 3.2 構成候補

配置空間と遮蔽に応じて次を比較する。

1. **一体型PCBアンテナ**
   - 部品数と故障点が少ない
   - 基板全体を外装裏へ配置できる場合に採用

2. **FPCアンテナ分離型**
   - 電池・SoCを深い隠蔽空間へ置き、薄いアンテナだけを担当方向寄りの非視認位置へ配置
   - 山笠の外形から突出させない

3. **同軸・FPC延長型**
   - 本体とアンテナの最適位置が離れる場合に限定
   - コネクター、損失、振動故障、組み付けを評価する

4. **方向別基板派生**
   - 4タグの論理設計を共通化しつつ、配置空間ごとにアンテナ位置や基板外形を変える

### 3.3 アンテナkeep-out

アンテナ周辺では以下を避ける。

- 銅箔
- 電池
- 金属ネジ、金属ブラケット
- LED配線束
- 大型金属装飾
- 吸水した布への密着

アンテナ形式と整合回路は、裸基板や筐体単体ではなく、山笠へ隠蔽した最終状態で調整する。

### 3.4 初期無線設定

| 項目 | 初期値 |
|---|---:|
| PHY | BLE 1M |
| 広告間隔 | 250ms |
| 送信出力 | 0dBm |
| 接続受付 | 無効 |
| 運用LED | 無効 |

試験結果により、タグ単位で-4、0、+4、+8dBmを選択する。全タグを最初から最大出力にせず、隠蔽状態で必要な最小出力を採用する。

## 4. 広告プロトコル

### 4.1 ペイロード

```text
protocol_version   1 byte
yamakasa_id        4 bytes
node_id            1 byte
boot_counter       2 bytes
sequence           2 bytes
battery_state      1 byte
auth_tag           8 bytes
```

合計19バイトを想定し、Manufacturer Specific Dataに収める。

### 4.2 ID

- `yamakasa_id`: 山笠を表す固定公開ID
- `node_id`: `front`, `rear`, `left`, `right`

山笠の位置は意図的に公開するため、IDローテーションは行わない。

### 4.3 認証

固定IDの秘匿ではなく、偽タグとリプレイの防止を目的とする。

```text
auth_tag = Truncate(MAC(
  tag_secret,
  protocol_version |
  yamakasa_id |
  node_id |
  boot_counter |
  sequence |
  battery_state
))
```

秘密鍵はタグごとに異なるものを使用し、サーバー側で管理する。アプリは広告を収集してサーバーへ渡し、最終認証をサーバーで行う。

### 4.4 シーケンス

`sequence`は5〜30秒程度ごとに増加させる。`boot_counter`は不揮発領域へ保存し、再起動ごとに増加させる。

## 5. ファームウェア設計

```text
BOOT
  ↓
設定・鍵・boot_counter読込
  ↓
広告ペイロード生成
  ↓
ADVERTISING
  ├── sequence更新
  ├── 電池状態更新
  ├── MAC再計算
  └── watchdog監視
```

通常運用では広告を停止しない。夜間に街中へ停車している期間も24時間広告を継続する。

電池状態は`GOOD`、`LOW`、`REPLACE`の段階値とする。watchdog、ブラウンアウト復旧、設定破損時の安全な広告を実装する。運用中のLED通知は行わない。

## 6. 筐体設計

### 6.1 設計原則

筐体は独立した見える製品ではなく、山笠の既存構造内へ完全に隠蔽される内部部品として設計する。

- IP67相当
- 電池交換式
- 低光沢の非導電性材料
- 外装面から見えない
- 山笠の外形から突出しない
- 人形の全可動域で露出・干渉しない
- 外部に見えるLED、ボタン、ロゴ、ラベルを持たない
- 固定部と脱落防止部を含めて隠蔽する
- 取り外し後に痕跡を残さない

### 6.2 形状

円形や正方形の共通筐体へ固定せず、利用可能な空間に応じて次を許容する。

- 薄型長方形
- 電池部とアンテナ部を細長く分離した形状
- 方向別に外形が異なる筐体
- 既存構造へ沿う低背形状

初期上限目安：

- 外形：30〜45mm
- 厚さ：10〜15mm
- 重量：10〜20g

これは目標寸法ではなく上限候補であり、隠蔽空間へ収まらない場合は小型化する。

### 6.3 固定

- 既存構造の裏側へ着脱式に固定する
- 外観部品へ穴あけ、切削、恒久接着を行わない
- 固定具と脱落防止具を外から見せない
- 電池交換と点検は裏側または既存点検口から行う

## 7. 4タグの論理統合

4タグは物理的には独立しているが、アプリとバックエンドでは1つの山笠として扱う。

```text
Yamakasa 001
├── front
├── rear
├── left
└── right
```

アプリはどれか1タグを有効に検出すれば、その山笠の位置送信候補になれる。複数タグの検出情報は、検出数、最終検出時刻、受信回数、RSSI中央値として1つの位置報告へまとめる。

## 8. 製造・設定

各タグへ以下を書き込む。

- protocol version
- yamakasa ID
- node ID
- tag secret
- initial boot counter
- RF calibration/configuration

前・後・左・右のラベル、QRコード、管理情報は、タグを取り外した場合または保守時だけ見える面に付ける。山笠の外観面へ恒久ラベルを付けない。

## 9. 検証と設計完了条件

実機検証で確定する事項：

- 一体型PCBアンテナかFPC分離アンテナか
- 各方向で利用できる完全非視認空間
- 各タグの最適送信出力
- 広告間隔250msが必要か、500msで十分か
- 筐体寸法、形状、固定方法
- 前後左右の最終取り付け位置
- LED点灯パターンによる受信率への影響

設計完了には次の両方を必要とする。

1. 完成状態の山笠で無線性能・防水・耐振動要件を満たす
2. 昼夜、全方向、人形全状態で機器が視認されず、山笠の意匠・制作・運営責任者の外観承認を得る

外観要件を満たさないハードウェアは、技術的に動作していても設計未完了とする。
