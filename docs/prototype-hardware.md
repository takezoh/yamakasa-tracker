# プロトタイプ・ハードウェア方針

## 1. 目的

最初のプロトタイプでは、山笠上でBLE中継方式が成立するかを、不要なRF設計要因を増やさずに検証する。

検証対象：

- 電飾ON時のBLE検出率
- 人形の全開・全閉・途中状態による方向別遮蔽
- 4タグによる死角補完
- iPhone／Androidのバックグラウンド検出率
- 必要な送信出力と広告間隔
- 外観を損なわない隠蔽配置の成立性

## 2. 初期プロトタイプ

初期段階は、一体型BLEモジュールまたは評価基板の内蔵PCBアンテナを使う。

```text
BLEモジュール
├── BLE SoC
├── 水晶
├── 整合回路
├── 内蔵PCBアンテナ
└── CR2032または試験用電源
```

SoC単体にアンテナが内蔵されているとは限らないため、ここでいう一体型は、アンテナと整合回路を含むモジュールまたは基板を指す。

## 3. 識別子

プロトタイプ段階から本番と同じ識別概念を使う。

- `resource_id`：追跡対象を表す64-bit固定公開ID
- `node_id`：`front`、`rear`、`left`、`right`

BLE広告では`resource_id`を8-byte big-endian binaryで送る。アプリ、API、ログでは同じ値を16桁の小文字16進文字列で表現する。

`yamakasa_id`、`beacon_id`などの別名は使わない。

## 4. 段階設計

### Phase 1：方式検証

市販モジュールまたは評価基板を使い、以下を測る。

- 電飾ON/OFF差
- 人形開閉差
- 前後左右・斜め方向の検出率
- 5m、10m、20mでの検出率
- 送信出力と広告間隔
- OS・機種差

この段階では最終サイズ、防水、量産性を確定しない。

### Phase 2：小型一体型タグ

方式成立後、CR2032、防水筐体、一体型アンテナを含む小型タグを作る。

- 外観から完全に隠蔽
- IP67相当
- 168時間以上動作
- 4タグ同一ハードウェア
- 違いは`node_id`だけ

### Phase 3：必要方向だけ分離アンテナ化

一体型では外観と無線性能を両立できない方向だけ、短い同軸付きFPCアンテナを採用する。

分離型への移行条件：

- 本体を隠せる位置とアンテナ最適位置が一致しない
- 毎年変わる装飾で一体型アンテナが遮蔽される
- 外観を保つと検出率目標を満たせない

一体型で合格した方向を分離型へ変更しない。

## 5. 毎年の再検証

山笠は毎年、人形と外装が変わるため、前年の配置を無条件に流用しない。

1. その年の装飾完成状態を確認
2. 4方向の候補位置へ一体型タグを仮設
3. 人形の全状態と電飾の全パターンで測定
4. 外観審査を実施
5. 一体型で不合格の方向だけ分離型を試験
6. 合格した位置、向き、出力、広告間隔を記録

## 6. プロトタイプ合格条件

- BLE広告の`resource_id`と`node_id`をアプリが正しく検出できる
- 10m以内で20秒以内の検出成功率99%以上
- 20m以内で20秒以内の検出成功率95%以上
- 人形の全開・全閉・途中状態で重大な死角がない
- 電飾最大点灯中でも検出率を維持する
- タグ、アンテナ、固定具が観覧者から見えない
- 一体型で成立する方向と、分離型が必要な方向を実測で判定できる

## 7. 非目標

初期プロトタイプでは以下を完成させない。

- 最終量産基板
- 最終防水筐体
- 最小サイズ
- 分離アンテナの全方向採用
- 毎年共通の固定配置

プロトタイプの目的は、方式、配置、無線条件、外観条件の成立性を反証可能な形で確認することである。
