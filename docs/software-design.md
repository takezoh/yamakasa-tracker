# ソフトウェア設計

## 1. 目的

山笠に搭載した4個のBLEタグを周囲の主催者スマートフォン群が検出し、端末間で位置取得と送信を分散する。個々の端末のバッテリー消費を抑えながら、観客向けWebへ山笠の現在位置、開催状態、進行方向を公開する。

優先順位は次の通り。

1. 位置精度
2. 端末の省電力
3. 位置更新の継続性
4. 開催・非開催判定の即時性

開催・非開催判定のタイミングは厳密でなくてよい。頻繁なpresence送信や競合確認のためのネットワークアクセスは避け、その分の電力を少数端末の高品質な測位へ使う。

暫定的に、有効端末数が10台以上なら`ACTIVE`、10台未満なら`INACTIVE`と判定する。

## 2. システム構成

```text
山笠BLEタグ群
    ↓ BLE広告
主催者モバイルアプリ群
    ├── BLE検出
    ├── ローカル送信候補判定
    ├── 選出端末だけ単発測位
    └── 少数の重複を許容して位置送信
            ↓ HTTPS
バックエンド
    ├── 端末presence集約
    ├── 開催状態判定
    ├── 位置レポート検証・統合
    ├── Google Roads APIによる道路スナップ
    ├── 進行方向推定
    └── 公開状態生成
            ↓ SSE / polling
観客向けWeb
    ├── Google Maps
    ├── 山笠の推定位置
    ├── 進行方向
    └── 開催状態・更新時刻
```

## 3. モバイルアプリ


### 3.1 タグ検出状態

4タグは同じ`yamakasa_id`と異なる`node_id`を持つ。アプリは4タグを1つの山笠として扱う。

```text
ABSENT
  ↓ 10秒以内に正規タグを2回以上検出
ELIGIBLE
  ↓ 90秒間検出なし
ABSENT
```

- 送信直前の必須条件：いずれかのタグを直近10秒以内に複数回検出
- RSSIは品質評価に使うが、単独で距離や参加状態を決めない
- タグ非検出中は位置を取得・送信しない

### 3.2 匿名端末ID

- `relay_session_id`はイベントまたは数時間単位で生成する
- 氏名、電話番号、広告IDなどの恒久識別子は送信しない
- 個人の移動履歴は保持しない

## 4. 送信端末の決定

### 4.1 方針

サーバーは毎回送信端末を指名しない。全候補端末が最新状態をGETして厳密な排他制御を行う方式も採用しない。

各端末がローカルで同じ規則を使い、自分がその時間スロットの送信候補かを判定する。厳密に1台へ絞るための通信より、1〜3台程度の重複送信を許容する方がシステム全体で省電力になる。

### 4.2 時間スロット

初期値は30秒とする。

```text
slot_id = floor(current_time / 30 seconds)
```

位置更新目標は30秒前後とし、実地試験後に20〜60秒で調整する。

### 4.3 決定論的な確率抽選

各端末はローカルで次を計算する。

```text
random_value = Hash(
  yamakasa_id |
  slot_id |
  relay_session_id
)
```

10台程度の参加を前提とする初期候補確率は20%とする。

```text
random_value が下位20% → 送信候補
それ以外                → 何もしない
```

10台なら平均2台、20台なら平均4台が候補になる。重複数が多すぎる場合は候補確率を10〜15%へ下げる。

参加台数を送信前に問い合わせる必要はない。参加台数推定は、設定更新時や低頻度presence応答時に得られた値を候補確率の補正に使ってよいが、必須にはしない。

### 4.4 端末状態による重み付け

次の条件では候補になりにくくする。

- 直近5〜10分以内に送信済み
- 低電力モード中
- バッテリー残量が少ない
- BLE検出が弱い、単発、または不安定
- ネットワーク状態が悪い
- 位置権限またはバックグラウンド権限が不足

次の条件では候補になりやすくする。

- 複数の山笠タグを継続検出
- 最近送信していない
- 充電中
- OSが新しく高精度な位置を保持している
- 通信状態が良い

実装は、候補確率の補正または抽選スコアへのペナルティとして行う。

### 4.5 候補内バックオフ

候補端末は0〜10秒の決定論的な待機時間を持つ。

```text
delay = Hash(slot_id | relay_session_id | "delay") mod 10 seconds
```

待機後に再度BLEタグを直近10秒以内に検出していることを確認する。サーバーへの事前GETは行わず、そのまま測位とPOSTへ進む。

## 5. 位置取得と省電力

### 5.1 キャッシュ位置

キャッシュ位置とは、OSが直前に別処理で取得し端末内に保持している位置である。新しくGNSSを起動せず利用できるため省電力だが、古さと精度を必ず評価する。

初期採用条件：

- 取得から20秒以内
- 水平精度20m以内
- BLEタグを直近10秒以内に複数回検出
- 前回公開位置から物理的に不自然でない

位置精度を優先するため、以前の50m基準より厳しくする。条件を満たさない場合は単発測位へ進む。

### 5.2 単発測位

候補端末だけが位置を取得する。

1. 厳しい条件を満たすキャッシュ位置があれば使用
2. なければ単発で高精度位置を要求
3. 最大8〜12秒待つ
4. 水平精度30m以内を通常採用
5. 30〜50mは他端末レポートとの統合候補
6. 50m超は原則不採用
7. 取得後は位置更新を直ちに停止

常時高精度測位、常時ナビゲーション、全端末の定期測位は採用しない。

### 5.3 省電力の原則

電力コストは概ね次の順で大きい。

1. 高精度位置の新規取得
2. セルラー通信の開始
3. バックグラウンド起動
4. BLE監視
5. ローカルの時刻・ハッシュ判定

そのため次を守る。

- 大多数の端末はBLE監視とローカル抽選だけ
- 候補端末だけが位置を評価
- 良質なキャッシュ位置がある場合だけ再利用
- 新規測位はさらに限定
- 競合確認GETは行わない
- 位置POSTとpresence更新を可能な限り同一通信へまとめる

## 7. 開催・非開催判定

### 7.1 方針

開催判定は即時性を求めず、省電力を優先する。30〜60秒ごとのpresence送信は行わない。

presenceは次のタイミングに限定する。

- BLEタグを新たに検出して候補群へ参加したとき
- 位置レポート送信時
- 前回通信から3〜5分経過したとき
- BLE検出状態や権限状態が大きく変わったとき

退出通知は必須とせず、リース期限切れで処理する。

### 7.2 有効端末

初期値：

- presenceリース：7分
- 定期更新目標：5分
- 同一`relay_session_id`の重複は1台として扱う
- 正規タグをpresence送信直前に検出していること

### 7.3 暫定判定

```text
active_device_count >= 10 → ACTIVE
active_device_count < 10  → INACTIVE
```

ヒステリシス：

- `INACTIVE → ACTIVE`：10台以上を2分継続
- `ACTIVE → INACTIVE`：10台未満を10分継続

開始・終了表示が数分遅れても許容する。位置公開処理は開催状態から分離し、開催判定の遅延が位置精度や位置更新を妨げないようにする。

## 8. バックエンド位置統合

### 8.1 検証

レポートごとに次を確認する。

- 正規アプリ・認証済みタグ
- BLE検出の鮮度
- 位置の取得時刻と精度
- 山笠の想定最高速度を超える不自然なジャンプ
- 同一端末からの異常頻度
- 同一スロット内の他レポートとの整合性

### 8.2 統合

同一スロットに複数レポートがある場合：

1. 精度50m超を除外
2. 時系列上不可能な位置を除外
3. 他レポート群から大きく外れる点を除外
4. 残った位置を精度と鮮度で重み付け
5. 最良点または重み付き中心を生推定位置とする

単一端末の精度表示をそのまま公開精度にせず、レポート間の分散も信頼度へ反映する。

## 9. Google Maps道路スナップ

### 9.1 処理

直近の生推定位置列をGoogle Roads APIのSnap to Roadsへ渡す。

- 単点ではなく直近3〜10点を使用
- 点列は時刻順にする
- `interpolate=true`は表示軌跡用途に限定
- API結果の`placeId`と道路形状を保持

### 9.2 採用条件

道路スナップ結果は無条件に採用しない。

- 元位置から道路までの距離が位置精度と整合する
- 前回採用道路と連続している、または合理的に接続できる
- 山笠の最高速度を超える道路ジャンプがない
- 平行道路がある場合、過去の進行経路との連続性を優先
- 道路外を通る正当な区間では生位置または平滑化位置へ戻す

Google Roads API障害時も位置公開を継続する。

## 10. 進行方向推定

進行方向はスマートフォン個別のheadingやbearingを直接採用しない。端末の向きや参加者個人の歩行方向は山笠の進行方向と一致しないためである。

道路補正済みの位置時系列から推定する。

1. 直近30〜90秒の採用位置を取得
2. 外れ値を除外
3. 道路上の移動距離が位置誤差を十分上回るか確認
4. 最古点から最新点への道路接線方向を算出
5. 過去方向と平滑化
6. 信頼度を付与

信頼度：

- `HIGH`：複数点が同一道路上で一貫し、移動距離が誤差を十分上回る
- `MEDIUM`：方向は推定可能だが位置分散がある
- `LOW`：移動距離が小さい、交差点、平行道路、位置誤差が大きい
- `UNKNOWN`：方向を表示しない

低信頼時は誤った矢印を出すより、方向表示を省略する。

## 11. API

```text
POST /v1/yamakasa/{id}/presence
POST /v1/yamakasa/{id}/reports
GET  /v1/public/yamakasa/{id}
GET  /v1/public/yamakasa/{id}/stream
```

送信候補決定のための`GET /relay-state`は削除する。

公開応答例：

```json

  "status": "ACTIVE",
  "position": {

  },

    "degrees": 82,
    "confidence": "HIGH"
  }
}
```

## 13. 検証指標

- 端末1台あたりの1時間の位置取得回数
- 端末1台あたりの1時間のネットワーク起動回数
- イベント1時間あたりの重複位置レポート数
- 位置精度50・90・95パーセンタイル
- 道路スナップ誤道路率
- 進行方向誤差と信頼度別精度
- 1時間あたりのバッテリー消費率
- iOS／Android、機種、低電力モード別の差

最適化対象は送信台数の最小化そのものではない。**観客に表示する位置精度を維持しながら、端末群全体と個々の端末の測位・通信・wake回数を最小化すること**を最終目的とする。
