# Raspberry Pi BLE技術検証プラン

## 1. 目的

手元にあるRaspberry Pi Zero W 1台とRaspberry Pi 4 Model B 1台をBLEタグとして使用し、専用ハードウェアを製作する前に、山笠位置中継方式の技術的成立性を確認する。

この検証の目的は最終タグの性能評価ではない。以下のソフトウェアおよび方式上のリスクを、追加ハードウェア購入前に除去することである。

- Linux端末から独自BLE広告を継続送信できる
- iOS／Androidアプリが広告を検出できる
- 同一`resource_id`を持つ複数タグを1つの追跡対象として統合できる
- 異なる`node_id`を識別できる
- BLE検出を位置取得・送信の前提条件として使用できる
- フォアグラウンドおよびバックグラウンドで必要な検出フローが成立する

## 2. 期待アウトカム

検証完了時に、次の判断を可能にする。

1. BLE中継方式を継続して専用小型タグへ進めるか
2. 広告フォーマットまたはアプリ検出方式を変更すべきか
3. iOS／Androidのバックグラウンド制約が方式上の阻害要因になるか
4. 4タグ構成の実地検証へ進むために必要な追加ハードウェアは何か

## 3. 検証範囲

### 3.1 検証するもの

- BLE広告の生成と継続送信
- `resource_id`、`node_id`、`sequence`の送受信
- 同一`resource_id`の複数タグ統合
- 一方のタグだけを検出した場合の動作
- 複数タグを同時検出した場合の品質評価
- BLE検出後の端末位置取得
- 位置レポート生成までのアプリ内フロー
- iOS／Androidのフォアグラウンド検出
- iOS／Androidのバックグラウンド検出
- アプリ再起動、画面消灯、一定時間放置後の再検出

### 3.2 検証しないもの

- CR2032で168時間動作するか
- 最終タグの消費電力
- 最終筐体サイズと隠蔽性
- IP67、防水、振動、衝撃耐性
- 最終アンテナの到達距離
- Raspberry PiのRSSIを最終タグのRF性能として評価すること
- 前後左右4タグによる完全な方向別遮蔽評価
- Raspberry PiをバックエンドまたはWebサーバーとして使うこと

## 4. 使用機材と役割

| 機材 | 役割 | 識別子 |
|---|---|---|
| Raspberry Pi Zero W | BLEタグA | `node_id = front` |
| Raspberry Pi 4 Model B | BLEタグB | `node_id = rear` |
| iPhone | 中継アプリ検証端末 | `relay_session_id`を生成 |
| Android端末 | 中継アプリ検証端末 | `relay_session_id`を生成 |

両タグは同じ`resource_id`を使用する。

```text
resource_id = 01af34c9812de704

Zero W:
  node_id = front

Pi 4 Model B:
  node_id = rear
```

Raspberry PiはBLE広告専用とし、バックエンド機能を兼務させない。

## 5. 最小広告フォーマット

最初の検証では、認証、電池情報、完全なリプレイ防止を実装せず、検出と統合に必要な最小フィールドだけを送る。

```text
resource_id: 8 bytes
node_id:     1 byte
sequence:    2 bytes
```

推奨値：

```text
resource_id: 01 af 34 c9 81 2d e7 04
node_id:
  front = 0x01
  rear  = 0x02
sequence:
  0〜65535を循環
```

技術成立後に、設計済みの`boot_counter`、`battery_state`、`auth_tag`を追加する。

## 6. 実装方針

### 6.1 Raspberry Pi側

- Raspberry Pi OS Liteを使用
- BlueZを使用して非接続BLE広告を送信
- Wi-Fiは初期設定とSSH作業にのみ使用し、検証中は不要なら無効化
- 起動後に広告プロセスを自動起動
- 広告間隔は可能な範囲で250ms相当を目標とする
- `sequence`を定期更新
- 広告停止時にsystemdで再起動

構成例：

```text
systemd
  └── yamakasa-ble-advertiser.service
        └── advertiser process
              ├── resource_id
              ├── node_id
              └── sequence
```

実装手段は、BlueZ D-Bus APIを優先する。`hcitool`など非推奨CLIへ恒久依存しない。

### 6.2 モバイルアプリ側

最小実装：

```text
BleScanner
  ↓
広告ペイロード解析
  ↓
resource_idでグループ化
  ↓
node_id別の最終検出時刻を保持
  ↓
TagPresenceTrackerをELIGIBLEへ遷移
```

初期画面に以下を表示する。

- 検出中の`resource_id`
- 検出中の`node_id`
- 各タグの最終検出時刻
- RSSI
- 受信した`sequence`
- アプリ状態：foreground／background
- 位置取得が実行されたか

## 7. 実行フェーズ

## Phase 1: Raspberry Pi BLE広告成立

### 作業

1. Zero WとPi 4へRaspberry Pi OS Liteを導入
2. Bluetoothアダプターを有効化
3. それぞれ異なる`node_id`を設定
4. BLE広告プロセスを起動
5. 汎用BLEスキャナーアプリで広告を確認
6. 再起動後に自動で広告が開始されることを確認

### 成功条件

- 両方のPiから広告を検出できる
- 同じ`resource_id`と異なる`node_id`を確認できる
- 30分連続で広告が停止しない
- 再起動後、手操作なしで広告が再開する

### 失敗時の判断

- 一方だけ失敗：PiまたはBlueZ設定固有の問題として切り分ける
- 両方失敗：広告実装またはペイロード構造を見直す

## Phase 2: モバイルアプリのフォアグラウンド検出

### 作業

1. iOS／Androidで広告スキャンを実装
2. `resource_id`と`node_id`を解析
3. 両タグを同一追跡対象へ統合
4. 片方ずつ停止し、残り1タグでも`ELIGIBLE`を維持できるか確認
5. 両タグ再開時に状態が回復するか確認

### 成功条件

- 両OSで2タグを識別できる
- 同じ`resource_id`として統合される
- 片方が停止しても追跡対象が消失しない
- 90秒以上すべてのタグを検出しない場合のみ`ABSENT`へ戻る
- 不正な`resource_id`または未知のペイロードを無視できる

## Phase 3: BLE検出と位置取得の接続

### 作業

1. タグ非検出時は位置を取得しないことを確認
2. 正規タグを直近10秒以内に複数回検出した場合のみ`ELIGIBLE`にする
3. 選出テスト用に候補判定を強制ONにできるデバッグ設定を設ける
4. 最終既知位置を取得
5. 条件不適合時に単発高精度測位を実行
6. 位置レポートをローカルログまたは開発用APIへ出力

### 成功条件

- タグ非検出中に位置取得が発生しない
- タグ検出後だけ位置取得へ進む
- レポートに`resourceId`と検出`nodeId`が入る
- タグ検出時刻と位置取得時刻を記録できる
- 一度の候補処理後に継続測位を停止する

## Phase 4: バックグラウンド成立性

### 試験状態

- アプリを前面表示
- ホーム画面へ移動
- 画面消灯
- 5分放置
- 15分放置
- アプリをOSによりサスペンドさせる
- 端末再起動後にアプリを起動して再試験

### 試験操作

1. タグを停止した状態でアプリをバックグラウンドへ移す
2. タグを起動する
3. 検出時刻を記録する
4. BLE検出後に位置取得・処理が実行できるか確認
5. 同じ試験をiOS／Androidで行う

### 成功条件

- OSが許可する範囲でバックグラウンド検出イベントを受け取れる
- 検出後の処理がOS制限時間内に完了する
- 位置取得またはレポート予約まで進める
- バックグラウンド制約と遅延を計測値として記録できる

本フェーズでは即時検出を要求しない。実運用で30秒更新を成立させられるかは、OS制約を含めて別途判定する。

## Phase 5: 遮蔽と距離の予備試験

Raspberry Piは最終タグと形状・アンテナ・電源が異なるため、結果を最終RF性能として扱わない。方式上の重大な欠落だけを確認する。

### 試験

- 見通し1m、5m、10m
- 木板の裏側
- 布の裏側
- 人体を間に置く
- LED配線付近
- Zero Wのみ検出可能な位置
- Pi 4のみ検出可能な位置

### 成功条件

- 一方のタグしか見えない状況でもアプリが追跡対象を維持する
- 遮蔽状態の違いを`node_id`別の検出ログで確認できる
- 完全欠落が起きる条件を再現可能な形で記録する

距離やRSSIの絶対値は専用小型タグの採否基準に使わない。

## 8. 計測項目

各試験で以下を記録する。

| 指標 | 内容 |
|---|---|
| detection_latency_ms | タグ起動から最初の検出まで |
| inter_detection_interval_ms | 同一タグの検出間隔 |
| missed_sequence_count | sequence欠落数 |
| detected_node_count | 同時に検出したタグ数 |
| last_seen_age_ms | 送信候補判定時のタグ鮮度 |
| app_state | foreground／background |
| location_requested | 測位要求の有無 |
| location_accuracy_m | 取得位置の水平精度 |
| processing_duration_ms | BLE検出から処理完了まで |
| device_model / OS | 端末とOSバージョン |

## 9. 完了条件

以下をすべて満たした時点で技術検証を完了とする。

- [ ] Zero WとPi 4が異なる`node_id`で安定広告する
- [ ] iOS／Androidが広告を解析できる
- [ ] 同一`resource_id`として統合できる
- [ ] 片方のタグだけでも追跡対象を維持できる
- [ ] タグ非検出時に測位しない
- [ ] BLE検出後に単発測位・レポート生成まで進める
- [ ] バックグラウンド検出の可否と遅延を両OSで計測した
- [ ] OS制約による阻害事項を記録した
- [ ] Raspberry Piの結果を最終RF・電池評価へ誤用しないことを確認した

## 10. 継続／中止判定

### 継続：小型BLEタグ4台へ進む

次を満たす場合：

- 両OSで広告を安定検出できる
- 同一対象への複数タグ統合が成立する
- BLE検出から位置処理へ接続できる
- バックグラウンド制約が、アプリ設計または運用で吸収可能

次段階：

```text
小型BLE開発基板 × 4
→ front / rear / left / right
→ 完成状態の山笠で隠蔽・遮蔽試験
→ CR2032専用基板
```

### 条件付き継続

以下の場合：

- iOSまたはAndroidの一方だけバックグラウンド動作が不安定
- 画面消灯後の検出遅延が大きい
- BLE検出後の高精度測位が時間内に完了しない

対応：

- 主催者による明示的なイベント参加開始
- フォアグラウンドサービス等のOS別設計
- 位置更新目標のOS別調整
- 役割端末の選出条件変更

### 中止または方式再設計

次の場合：

- 両OSでBLE広告を実用的に検出できない
- バックグラウンドでBLE検出から位置処理へ全く接続できない
- 必要な権限がストア配布要件と両立しない
- BLE検出を山笠近傍性の条件として信頼できない
